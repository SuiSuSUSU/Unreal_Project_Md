# Codex 작업 흐름

이 문서는 앞으로 Codex를 어떻게 사용할지 정리한다.

## 역할 분담

- Codex: 프로젝트 폴더의 Markdown 문서와 코드 파일을 읽고 수정한다.
- Codex: 작은 기능 단위 구현, 오류 수정, 빌드 확인, 변경 요약을 담당한다.
- ChatGPT: 기획 정리, 설계 검토, Codex에게 줄 지시문 작성에 활용한다.
- 개발자: 최종 판단, 3D 에셋 제작, 에디터 배치, 플레이 감각 확인을 담당한다.

## 기본 작업 규칙

- 큰 작업보다 작은 기능 단위로 요청한다.
- 구현 전 관련 Markdown 문서를 먼저 참고하게 한다.
- 확정된 내용과 후보 아이디어를 섞지 않는다.
- 구현 후 Codex에게 수정 파일, 테스트 방법, 다음 작업을 요약하게 한다.
- 새 C++ 클래스나 헤더 변경이 있으면 에디터 재시작 또는 전체 빌드가 필요할 수 있음을 고려한다.

## 좋은 Codex 지시문 예시

```text
Docs 문서를 먼저 읽고, 현재 확정된 조건을 벗어나지 않는 범위에서
SurvivorSquareEnemy가 플레이어 위치를 향해 이동하도록 수정해줘.
수정 후 변경 파일과 에디터에서 테스트하는 방법을 요약해줘.
```

```text
현재 스폰 로직을 유지하되, SpawnInterval이 시간이 지날수록 조금씩 줄어드는 기능만 추가해줘.
새로운 큰 시스템은 만들지 말고, Details 패널에서 조절 가능한 변수로 노출해줘.
```

```text
빌드 오류 로그를 보고 원인을 설명한 뒤, 필요한 최소 코드만 수정해줘.
수정 후 어떤 명령으로 검증했는지도 알려줘.
```

## 나쁜 Codex 지시문 예시

```text
게임 전체를 뱀파이어 서바이벌처럼 완성해줘.
```

문제가 되는 이유: 범위가 너무 크고, 장르가 아직 확정되지 않았으며, 기능 단위가 분리되어 있지 않다.

```text
재미있게 알아서 만들어줘.
```

문제가 되는 이유: 판단 기준이 없어서 구현 방향이 흔들린다.

```text
전투, 성장, UI, 보스, 저장, 스테이지를 한 번에 만들어줘.
```

문제가 되는 이유: 1인 개발과 Codex 의존 개발 기준으로 너무 큰 작업이다.

## 구현 후 확인할 항목

- 어떤 파일이 생성 또는 수정되었는가
- 에디터에서 어떤 액터나 블루프린트를 배치해야 하는가
- 어떤 값이 Details 패널에 노출되었는가
- 빌드 또는 Live Coding이 성공했는가
- 다음에 가장 작은 단위로 이어갈 작업은 무엇인가
