## 2026-07-04 Update: Confirmed-Hit Blood VFX

- `PlayerMeleeAttackSubsystem` now spawns blood VFX only after `HealthComponent::ApplyDamage` returns applied damage above zero.
- Current blood VFX source is `/Game/sA_SplatterofbloodFx/Fx/PreShaded_Blood_Alternatives`.
- The current implementation randomly chooses one of `NS_PreShadedBlood_1` through `NS_PreShadedBlood_10`.
- This is presentation-only hit feedback. It does not change damage, attack range, enemy death, stagger, hit react, or reward logic.
- If hit feedback becomes more complex, move it toward a dedicated hit-impact VFX owner rather than growing basic attack logic.

## 2026-07-02 Update: Weapon Impact VFX Boundary

- Basic attack hit detection and damage application remain in `PlayerMeleeAttackSubsystem`.
- Future `WeaponImpactVfxComponent` should only receive a confirmed-hit request after damage logic finds a valid target.
- Hit-impact VFX should be presentation-only. It must not decide whether an attack hit, how much damage is applied, or whether an enemy reacts.
- Weapon trails and hit-impact VFX are separate: `WeaponTrailComponent` owns sword trail playback, while future `WeaponImpactVfxComponent` owns contact bursts, sparks, blood, lightning contact, or ground-impact effects.
- Ground-contact VFX should use AnimNotify or an explicit ground trace when possible, not a fixed delay guessed from the attack start.
## 2026-07-01 Update: Weapon Trail Timing Rule

- Formal weapon-trail timing should be controlled by `AnimNotifyState_WeaponTrail` windows placed directly on the Combo1/Combo2/Combo3 attack animations.
- `WeaponTrailComponent.ComboTrailSlots` should primarily define which Niagara system is used and how it is attached, positioned, rotated, and scaled.
- `ActivationDelay` and `Duration` remain only as fallback timing values for attack-start playback when `bUseAnimNotifyTrailTiming` is disabled.
- In the editor these values are exposed as `Fallback Activation Delay` and `Fallback Duration`.
- When `bUseAnimNotifyTrailTiming` is enabled, `PlayAttackTrail()` does not start fallback playback and `AnimNotifyState_WeaponTrail` owns the on/off timing.
- When `bUseAnimNotifyTrailTiming` is disabled, `AnimNotifyState_WeaponTrail` does not start the trail, so fallback timing and notify timing do not overlap.
